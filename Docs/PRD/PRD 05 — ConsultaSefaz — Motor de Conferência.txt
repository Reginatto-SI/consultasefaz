# PRD 05 — ConsultaSefaz — Motor de Conferência

## 1. Objetivo
Definir o algoritmo determinístico de conferência entre base SEFAZ, base ERP e exceções.

## 2. Fonte de verdade deste PRD
Este documento é a fonte de verdade do processamento, da ordem das regras e da geração do resultado final.

## 3. Entradas
- notas_sefaz (com empresa_id e campos estruturais já resolvidos na importação)
- registros_erp (sem empresa)
- excecoes_ativas (empresa_id + chave_nfe)

Contrato detalhado: PRD 07.
Contrato de layout de origem SEFAZ: PRD 08.

Regra obrigatória: o motor não deve interpretar layout bruto de planilha; deve consumir somente campos estruturais entregues pela importação (PRD 02/08).

## 4. Pré-processamento obrigatório
- Normalizar chave/IE.
- Descartar registros sem chave válida (com log operacional).
- Indexar ERP para busca rápida.

## 5. Regras de classificação

**Matching segue regra oficial definida no PRD 07.**


### 5.0 Regra de normalização do status SEFAZ (V1)
- `status_sefaz_valido` representa nota autorizada/regular no relatório SEFAZ.
- `status_sefaz_invalido` representa nota cancelada, denegada, inutilizada, inexistente ou equivalente no relatório SEFAZ.
- A V1 deve tratar os textos reais do relatório SEFAZ de forma normalizada antes da classificação.
- Se aparecer status SEFAZ desconhecido, registrar aviso operacional e tratar como inválido por segurança, salvo regra futura explícita em contrário.

### 5.1 Regra de matching antes da classificação
Para cada nota SEFAZ:
1) Verificar exceção ativa.
2) Verificar status SEFAZ.
3) Buscar no RFT006/ERP por `chave_acesso = chave_nfe`.
4) Se encontrar a chave, validar se existe algum registro com `inscricao_estadual_emitente = emitente_inscricao_estadual`.

### 5.2 Resultado de matching (indicador oficial)
Usar o campo `resultado_matching` com os seguintes valores:
- `CONFIRMADO`
- `IE_EMITENTE_DIVERGENTE`
- `CHAVE_NAO_ENCONTRADA`

Regras de preenchimento:
- `CONFIRMADO`: chave encontrada e IE do emitente correspondente.
- `IE_EMITENTE_DIVERGENTE`: chave encontrada, sem correspondência da IE do emitente.
- `CHAVE_NAO_ENCONTRADA`: chave ausente no RFT006.

### 5.3 Matriz oficial de classificação final (V1)
| Condição | status_final |
|---|---|
| Exceção ativa | DESCONSIDERADA |
| Status SEFAZ inválido e resultado_matching = CONFIRMADO | IRREGULAR |
| Status SEFAZ inválido e resultado_matching = IE_EMITENTE_DIVERGENTE | IRREGULAR |
| Status SEFAZ inválido e resultado_matching = CHAVE_NAO_ENCONTRADA | IGNORADA |
| Status SEFAZ válido e resultado_matching = CONFIRMADO | OK |
| Status SEFAZ válido e resultado_matching = IE_EMITENTE_DIVERGENTE | IRREGULAR |
| Status SEFAZ válido e resultado_matching = CHAVE_NAO_ENCONTRADA | FALTANTE |

Observações obrigatórias:
- Status `IGNORADA` permanece interno e não deve aparecer na tabela principal da UI.
- `IGNORADA` pode ser usada internamente para auditoria técnica curta ou contagem operacional interna, mas não compõe os cards principais da V1.
- Status `IRREGULAR` cobre também escrituração encontrada com IE do emitente divergente.
- Não criar novo status final `DIVERGENTE` na V1.
- Não usar IE do destinatário para matching com RFT006.

## 6. Determinismo
- Mesma entrada produz mesma saída.
- Resultado não depende da ordem de arquivos.
- Sem aleatoriedade.

## 7. Disparos de execução
- Após importação SEFAZ.
- Após importação ERP.
- Após criação/reversão de exceção.

## 8. Saída
Dataset final por nota contendo status final, indicadores de matching e payload para UI.
Contrato de saída: PRD 07.

## 9. Limites V1
- Sem histórico completo de execuções.
- Sem processamento distribuído.
- Sem validações fiscais financeiras avançadas.
