# PRD 07 — ConsultaSefaz — Contrato de Dados e Pipeline Estrutural

---

## 1. Objetivo

Referência obrigatória de nomenclatura: PRD 00 — Dicionário de Domínio.

Definir o contrato estrutural mínimo do sistema para garantir consistência entre importação, conferência, exceções, UI e logs.

Este PRD é a fonte de verdade para:
- entidades mínimas;
- campos obrigatórios;
- pipeline oficial ponta a ponta;
- dataset final exibido na interface.

---

- Dicionário de domínio e nomenclatura: **PRD 00**

## 3. Princípios obrigatórios da V1


## Regra crítica de modelagem (OBRIGATÓRIO)
- `empresa_id` NÃO representa tenant.
- Representa destinatário fiscal.
- Qualquer evolução futura deve separar:
  - `tenant_id` (organização)
  - `destinatario_id` (fiscal)

## 3.1 Convenção semântica para V1
- No contexto operacional, entidade "Empresa" equivale a **Destinatário SEFAZ** (CNPJ destinatário da NF-e).
- `empresa_id` é mantido como nome de campo técnico por compatibilidade, com semântica de identificador do destinatário fiscal.
- ERP não define destinatário; ERP apenas confirma existência de nota da SEFAZ.


- Sistema multiempresa desde o início.
- SEFAZ é fonte única de verdade.
- Destinatário da nota é definido somente pelo SEFAZ (CNPJ destinatário).
- ERP nunca define destinatário.
- Exceções sobrescrevem qualquer regra automática.
- Processamento determinístico (mesma entrada = mesmo resultado).
- Modelo snapshot (sem histórico completo de execuções).

## 5. Pipeline oficial (V1)

1. Upload de arquivos (PRD 02)
2. Leitura, detecção de cabeçalho e colunas (PRD 02, PRD 08 e PRD 09)
3. Normalização de campos críticos (PRD 02 e PRD 05)
4. Snapshot das entradas: no SEFAZ, identificar os destinatários presentes no arquivo por `destinatario_cnpj_cpf` e substituir somente as notas desses destinatários impactados; destinatários ausentes no arquivo mantêm seus dados SEFAZ. No ERP/RFT006, snapshot global da base complementar vigente. (PRD 01/02)
5. Aplicação do motor determinístico (PRD 05)
6. Aplicação de exceções com prioridade máxima (PRD 04/05)
7. Geração do dataset final para UI (PRD 03/05)
8. Geração de métricas e resumo (PRD 03/05)
9. Registro de erros/avisos operacionais (PRD 06)

---

## 6. Regras operacionais de consistência

- Toda nota exibida deve ter origem em nota SEFAZ.
- Nunca criar nota a partir de ERP isolado.
- Exceção órfã (sem nota SEFAZ no snapshot atual) permanece cadastrada e não quebra o processamento.
- Exceção órfã não deve gerar status na tabela principal enquanto a nota não reaparecer na SEFAZ.
- Mudança em exceção ativa deve disparar nova conferência do snapshot atual.

---

## 7. Limites operacionais sugeridos para V1

- Processamento síncrono e em memória para volumes médios.
- Sem fila distribuída.
- Sem histórico completo de snapshots.
- Histórico curto apenas para logs operacionais (PRD 06).

---

## 8. Premissas de performance (V1)

- Normalização linear.
- Indexação ERP por chave de busca.
- Evitar laços quadráticos.
- Resultado previsível sem dependência da ordem dos arquivos.

## 10. Regra Oficial de Matching (V1)

Regra única e obrigatória para o motor de conferência:

1. Para cada nota SEFAZ, buscar `chave_acesso = chave_nfe` no RFT006.
2. Se a chave existir, validar se há registro com `inscricao_estadual_emitente = emitente_inscricao_estadual`.
3. Definir indicadores de matching:
   - Se chave + IE conferem:
     - `chave_existe_no_erp = true`
     - `ie_emitente_confere = true`
     - `encontrada_no_erp = true`
     - `resultado_matching = CONFIRMADO`
     - `motivo_divergencia = null`
   - Se chave existe e IE diverge:
     - `chave_existe_no_erp = true`
     - `ie_emitente_confere = false`
     - `encontrada_no_erp = false`
     - `resultado_matching = IE_EMITENTE_DIVERGENTE`
     - `motivo_divergencia = IE_EMITENTE_DIVERGENTE`
   - Se chave não existe:
     - `chave_existe_no_erp = false`
     - `ie_emitente_confere = false`
     - `encontrada_no_erp = false`
     - `resultado_matching = CHAVE_NAO_ENCONTRADA`
     - `motivo_divergencia = CHAVE_NAO_ENCONTRADA`
4. Nunca usar IE do destinatário para matching com RFT006.
5. Nunca exigir correspondência 1:1 entre linhas SEFAZ e ERP.
6. ERP nunca cria registros novos; ERP apenas confirma existência de nota da SEFAZ.

Restrições explícitas da V1:
- não usar heurística;
- não usar similaridade;
- não validar por nome, razão social, CNPJ de emitente/destinatário.



## 11. Integração explícita com o PRD 08 e PRD 09

- O PRD 08 define o contrato oficial de layout de entrada do relatório SEFAZ na V1.
- O campo técnico `empresa_id` permanece legado, mas representa o destinatário fiscal obtido do bloco `DADOS DESTINATÁRIO` do layout SEFAZ.
- No layout base da V1, `destinatario_cnpj_cpf` deve ser resolvido pela coluna `O` (`DADOS DESTINATÁRIO` + `CNPJ/CPF`), nunca pelo bloco de emitente.
- `payload_completo` deve manter integralmente os dados de entrada do relatório SEFAZ para rastreabilidade operacional.

- O PRD 09 define o layout oficial do relatório RFT006/ERP e os campos estruturais de confirmação de escrituração.


## 11. Esclarecimentos mandatórios de arquitetura da V1
- As entidades deste PRD são contratos de dados para memória e/ou armazenamento local do navegador; não são tabelas de banco na V1.
- `id`, `empresa_id`, `importacao_id` e `referencia_execucao` são identificadores locais gerados no navegador.
- É proibido criar migrations e tabelas na V1.
- O pipeline oficial roda localmente no navegador.
- O snapshot é local.
- O payload completo das importações permanece no estado local.
- A V1 não possui histórico completo nem persistência corporativa em servidor.
- Em evolução futura (V2+), estes contratos podem ser convertidos em modelo persistente, fora do escopo da V1.
