# PRD 00 — ConsultaSefaz — Dicionário de Domínio

## 1. Objetivo
Definir a fonte oficial de nomenclatura e semântica do ConsultaSefaz para eliminar ambiguidades entre termos de negócio e termos técnicos legados.

## 2. Fonte de verdade deste PRD
Este documento é a fonte de verdade para vocabulário de domínio e convenções semânticas.

## 3. Prioridade de governança
Em caso de conflito conceitual entre documentos, prevalece:
**PRD 00 > PRD 01 > PRD 07 > demais PRDs**.

Regra obrigatória: qualquer novo PRD deve respeitar este dicionário.

## 4. Conceitos oficiais do domínio (V1)

### 4.1 Tenant (Organização / Escritório)
- É o usuário organizacional do sistema (ex.: escritório contábil).
- Não está implementado formalmente na V1.
- Não deve ser confundido com destinatário fiscal.

### 4.2 Destinatário SEFAZ (conceito principal)
- É o CNPJ destinatário da NF-e no relatório SEFAZ.
- Define a quem a nota pertence no sistema.
- É a base de agrupamento operacional da V1.
- Substitui semanticamente o uso ambíguo de “empresa” na interface.

### 4.3 Emitente
- É o fornecedor/remetente da nota fiscal.
- Nunca define agrupamento principal do sistema.
- Pode aparecer em colunas e detalhes da nota.

### 4.4 ERP
- Fonte complementar de confirmação.
- Nunca define destinatário.
- Apenas confirma existência da nota SEFAZ conforme regras do motor.

### 4.5 Nota SEFAZ
- Entidade principal de conferência.
- Sempre vinculada a um destinatário SEFAZ.
- Origem obrigatória do dataset principal exibido na UI.

### 4.6 Exceção
- Regra manual com precedência sobre classificação automática.
- Sempre vinculada a destinatário + chave_nfe.

## 5. Regras obrigatórias de nomenclatura

### 5.1 UI
- Usar sempre: **Destinatário** ou **Destinatário SEFAZ**.
- Não usar “Empresa” como rótulo funcional na interface.

### 5.2 Backend / contratos técnicos (V1)
- Manter `empresa_id` e correlatos por compatibilidade da V1.
- Semântica oficial: `empresa_id` representa o identificador do destinatário fiscal.

### 5.3 Futuro (V2)
- Planejar migração de nomenclatura técnica:
  - `empresa_id` → `destinatario_id`
  - `empresa_nome` → `destinatario_nome`

## 6. Tabela de mapeamento

| Termo Antigo | Termo Correto | Contexto |
|--------------|----------------|----------|
| Empresa | Destinatário | UI |
| empresa_id | destinatario_id (futuro) | Backend |
| empresa_nome | destinatario_nome (futuro) | Backend |
| Multiempresa | Multidestinatário | Conceito |
| Empresa da nota | Destinatário SEFAZ da nota | Regra de negócio |

## 7. Aplicação prática imediata
- PRDs devem referenciar este dicionário ao tratar nomenclatura de domínio.
- Materiais de produto (README e interface) devem evitar ambiguidade entre tenant, destinatário e emitente.
- Mudanças desta natureza não alteram lógica de conferência, banco ou backend nesta etapa.


## 8. Regras de Blindagem de Domínio (OBRIGATÓRIO)
- `empresa_id` é **exclusivamente** identificador técnico de destinatário fiscal.
- É **proibido** interpretar `empresa_id` como tenant, organização ou usuário do sistema.
- O sistema NÃO implementa multi-tenant na V1.
- Qualquer futura implementação de multi-tenant deve utilizar:
  - `tenant_id` separado
  - sem reutilizar `empresa_id`
- Nenhuma regra de autenticação, usuário ou permissão deve depender de `empresa_id`.

## 9. Regra de Ouro do Projeto
- O sistema é **multidestinatário**, não multiempresa no sentido organizacional.
- Toda lógica principal gira em torno do **destinatário SEFAZ (CNPJ)**.
- SEFAZ define o destinatário. ERP nunca define.


## 10. Diretriz de arquitetura da V1 (obrigatória)
- A V1 não possui banco de dados.
- A V1 não possui backend.
- A V1 não possui autenticação.
- A V1 não possui tenant real.
- `empresa_id` continua como identificador técnico local do destinatário fiscal, gerado no navegador.
- `empresa_id` não representa tenant, organização, usuário nem registro persistente em banco.
- “Multidestinatário” na V1 significa processar múltiplos CNPJs destinatários dentro dos arquivos importados, com estado em memória e/ou armazenamento local do navegador.


## 11. Destinatários automáticos e exceções locais (V1)
- Destinatário fiscal na V1 é identificado automaticamente na importação SEFAZ pelos campos `destinatario_cnpj_cpf` e `destinatario_razao_social`.
- Destinatário não é tenant e não depende de cadastro manual em banco de dados.
- A tela de destinatários da V1 é de consulta/listagem local dos destinatários encontrados na importação.
- Exceção local é uma regra salva no navegador para desconsiderar uma chave específica de NF-e, com precedência sobre o motor automático.


## 12. Destinatário conhecido local (V1)
Definição:
- Destinatário conhecido local é um registro fixo no código da V1 para destinatários recorrentes.
- Pode conter dados operacionais como: apelido, razão social, CPF/CNPJ, IE, perfil e tributação.

Regras obrigatórias:
- Não é banco de dados.
- Não é cadastro corporativo.
- Não é mock de demonstração.
- Não define pertencimento fiscal da nota.
- Apenas enriquece a exibição quando o CPF/CNPJ da SEFAZ coincide com a base local.
